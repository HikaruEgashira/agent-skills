# Seeded-Fault Benchmark

falsehood-audit の recall / precision を、グラウンドトゥルースが存在しない任意のリポジトリで測定する手順。既知の虚偽を注入した複製を作り、ブラインド監査の検出結果を機械採点する。

## 手順

### 1. 複製と基線確認

対象リポジトリを作業ディレクトリに複製する。複製に対して監査を1回実行し、注入前から存在する finding を記録する(基線 finding は採点から除外する)。

### 2. 注入

seeder エージェント(監査エージェントとは別インスタンス)が、型ごとに以下の規則で虚偽を注入する。各注入について `seeded.json` に 型 / ファイル / 位置 / 内容 / 検出に必要な検査ID を記録する。注入は既存テスト・CI をすべて通過する状態を維持すること(通過しない注入は虚偽ではなく単なるバグであり、監査対象として不適)。

| 型 | 注入規則 |
|----|----------|
| 0 | 実在しない識別子(パッケージ名・型番・API名・引用)への参照を文書または設定に追加する |
| A | 単位換算・符号化を1箇所、形式的に有効なまま意味的に誤らせる(切り捨て/切り上げの交換、単位系の混同、フィールド上限を超える値) |
| B | 2つの成果物が共有する量・能力の一方だけを変更する(宣言側と消費側、要件と制約、図と表) |
| C | 操作可能要素(UIの選択肢・CLIフラグ・設定キー)を1つ追加し、どのデータフローにも接続しない |
| D | テスト1つを、実装を呼ばずに固定値をassertする形に書き換える。またはCIから検証ステップを1つ外し、文書の検証済み主張は残す |
| E | 公開表面(CLIサブコマンド・APIシンボル)を1つ追加し、文書に記載しない |

型ごとに最低1件、合計6件以上。注入箇所は互いに独立させる(1つの発見が別の注入を露出させない)。

### 3. ブラインド監査

seeder とは別の監査エージェント(seeded.json への読み取りアクセスなし)が、複製に対して falsehood-audit を実行する。分割実行モードの効果を測る場合は、単独実行と分割実行の両方を同一の複製に対して実行する。

### 4. 採点

`seeded.json` の各注入について、監査レポートの findings と突合する:

- **検出**: finding が注入箇所を特定し、型分類が一致または隣接(A/B の境界事例は一致扱い)
- **部分検出**: 注入箇所は特定したが型または根因の説明が誤り
- **未検出**: findings に対応が無い

`grading.json` に全注入の判定と根拠(レポートの該当行の引用)を記録する。加えて、注入と無関係な finding を基線 finding と突合し、どちらにも無いものを false positive 候補として人手検証する。

### 5. 集計

`benchmark.json` に記録する:

```json
{
  "target_repo": "<name>",
  "config": "single | split",
  "seeded": 6,
  "detected": 0,
  "partial": 0,
  "missed": 0,
  "recall": 0.0,
  "false_positive_candidates": 0,
  "tokens_total": 0
}
```

構成間(単独/分割、スキルバージョン間)の比較は recall の差分と tokens_total の比で行う。

## 制約

- seeder と監査エージェントのセッション・コンテキストを共有しない
- 注入内容をスキル本文・エージェント指示に書かない(検査ルールの汚染)
- 同一複製の再監査は注入位置を変えて行う(位置の記憶による過大評価を防ぐ)
