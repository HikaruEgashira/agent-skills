## 1. Writing the description

The description is a skill's only trigger mechanism. Claude decides whether to use a skill from just the name + description in the `available_skills` list.

### How the trigger works

Claude tends not to invoke a skill for simple tasks its built-in tools handle easily. "Read this PDF" may not trigger even with a flawless description. The more complex, multi-step, and specialized the task, the higher the trigger probability.

### Principles

1. State both **what the skill does** and the **concrete situations that trigger it**
2. Spell out the boundary conditions that distinguish a near-match that should *not* trigger
3. Lean slightly "pushy" — to compensate for Claude's conservative trigger judgment

### Good examples

```yaml
description: "Perform any PDF task — read files, extract text/tables, merge, split,
  rotate, watermark, encrypt/decrypt, OCR. Use this skill whenever a .pdf file is
  mentioned or a PDF deliverable is requested. Especially useful when the task is
  conversion/editing/analysis rather than a plain 'read it' request."
```

```yaml
description: "Any spreadsheet task on Excel/CSV/TSV files — adding columns, formulas,
  formatting, charts, data cleaning. Use this skill whenever the user mentions a
  spreadsheet file, even casually ('the xlsx in my downloads')."
```

### Bad examples

- `"A skill that processes data"` — too vague; unclear which files/tasks
- `"PDF-related work"` — no concrete actions listed, no trigger situations

## 2. Body writing style

### Why-first

When an LLM understands the reason, it judges correctly even in edge cases. Conveying context beats coercive rules.

**Bad:**
```markdown
ALWAYS use pdfplumber for table extraction. NEVER use PyPDF2 for tables.
```

**Good:**
```markdown
Use pdfplumber for table extraction. PyPDF2 specializes in text extraction and
doesn't preserve a table's row/column structure, whereas pdfplumber recognizes
cell boundaries and returns structured data.
```

### Generalize

When feedback or test results surface a problem, **generalize at the level of principle** instead of a narrow fix that only suits the specific example.

**Overfit fix:**
```markdown
If there is a "Q4 revenue" column, convert that column to numbers.
```

**Generalized fix:**
```markdown
If a column name contains a keyword implying a number — "revenue", "amount",
"quantity", etc. — convert that column to a numeric type. Keep the original value
if conversion fails.
```

### Imperative voice

Write "do X" / "use X", not "this can do X" / "you may want to". A skill is an instruction sheet.

### Spend context sparingly

The context window is a commons. Ask whether each sentence justifies its token cost:
- "Does Claude already know this?" → cut it
- "Does Claude make a mistake without this?" → keep it
- "Would one concrete example beat a long explanation?" → replace with the example

## 3. Output format patterns

For skills where the deliverable's format matters:

```markdown
## Report structure
Follow this template exactly:

# [Title]
## Summary
## Key findings
## Recommendations
```

Keep format definitions concise; a real example makes them more effective.

## 4. Example patterns

Examples beat long explanations:

```markdown
## Commit message format

**Example 1:**
input: Add JWT-token-based user authentication
output: feat(auth): implement JWT-based authentication

**Example 2:**
input: Fix a bug where the show-password button doesn't work on the login page
output: fix(login): fix show-password toggle button
```

## 5. Progressive Disclosure patterns

### Pattern 1: split by domain

```
bigquery-skill/
├── SKILL.md (overview + domain-selection guide)
└── references/
    ├── finance.md (revenue, billing metrics)
    ├── sales.md (opportunities, pipeline)
    └── product.md (API usage, features)
```

When the user asks about revenue, only finance.md loads.

### Pattern 2: conditional detail

```markdown
# DOCX processing

## Creating a document
Create a new document with docx-js. → see [DOCX-JS.md](references/docx-js.md).

## Editing a document
For simple edits, modify the XML directly.
**If you need tracked changes**: see [REDLINING.md](references/redlining.md)
```

### Pattern 3: structuring large reference files

A reference file over 300 lines carries a table of contents at the top:

```markdown
# API Reference

## Contents
1. [Authentication](#authentication)
2. [Endpoint list](#endpoint-list)
3. [Error codes](#error-codes)
4. [Rate limits](#rate-limits)

---

## Authentication
...
```

## 6. Deciding what to bundle as a script

Observe the agents' transcripts across test runs. These signals mark a bundling candidate:

| Signal | Action |
|------|------|
| The same helper script generated in 3 of 3 tests | Bundle it into `scripts/` |
| The same pip install/npm install every time | State a dependency-install step in the skill |
| The same multi-step approach repeated | Document it as a standard procedure in the skill body |
| The same workaround applied after a similar error every time | Document the known issue and its fix in the skill |

A bundled script must pass an execution test.

## 7. Data schema standards

Use standard schemas for consistent data exchange between skills. They apply to testing/evaluating the skills Harness generates.

### eval_metadata.json

Metadata for each test case:

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "the user's task prompt",
  "assertions": [
    "the deliverable contains X",
    "a file was created in Y format"
  ]
}
```

### grading.json

Assertion-based scoring result:

```json
{
  "expectations": [
    {
      "text": "Deliverable contains 'Seoul'",
      "passed": true,
      "evidence": "Confirmed 'Seoul region data extraction' at step 3"
    }
  ],
  "summary": {
    "passed": 2,
    "failed": 1,
    "total": 3,
    "pass_rate": 0.67
  }
}
```

**Field names matter:** use `text`, `passed`, `evidence` exactly (no variants like `name`/`met`/`details`).

### timing.json

Execution time/token measurements:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

Save `total_tokens` and `duration_ms` from the subagent completion notification immediately. This data is reachable only at notification time and cannot be recovered later.

## 8. What not to include

- Auxiliary docs like README.md, CHANGELOG.md, INSTALLATION_GUIDE.md
- Meta-information about the skill's creation (test results, iteration history)
- User-facing manuals (a skill is an instruction sheet for an AI agent)
- General knowledge Claude already has
