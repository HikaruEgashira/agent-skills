## 1. Testing framework overview

Skill quality verification combines **qualitative** and **quantitative** evaluation.

| Type | Method | Fits which skills |
|----------|------|-----------|
| **Qualitative** | The user reviews the deliverable directly | Subjective quality: writing style, design, creative work |
| **Quantitative** | Automated assertion-based scoring | Objectively verifiable: file creation, data extraction, code generation |

The core loop: **write → run test → evaluate → improve → re-test**

## 2. Writing test prompts

### Principle

A test prompt should read like a **specific, natural sentence a real user would type**. Abstract or artificial prompts have little testing value.

### Bad

```
"Process the PDF"
"Extract the data"
"Generate a chart"
```

### Good

```
"In 'Q4_revenue_final_v2.xlsx' in my downloads folder, add a profit-margin (%)
column using column C (revenue) and column D (cost). Then sort descending by margin."
```

```
"Extract the table on page 3 of this PDF and convert it to CSV. The header
spans two rows — the first is the category, the second is the actual column name."
```

### Prompt diversity

- Mix **formal / casual** tone
- Mix **explicit / implicit** intent (file format stated outright vs. inferred from context)
- Mix **simple / complex** tasks
- Include some abbreviations, typos, and casual phrasing

### Coverage

Start with 2–3 prompts, designed to cover:
- one core use case
- one edge case
- (optional) one compound task

## 3. Execution test: with-skill vs. baseline

### 3-1. Comparison structure

For each test prompt, spawn two subagents **simultaneously**:

**With-skill run:**
```
prompt: "{test prompt}"
skill path: {skill path}
output path: _workspace/iteration-N/eval-{id}/with_skill/outputs/
```

**Baseline run:**
```
prompt: "{test prompt}"  (identical)
skill: none
output path: _workspace/iteration-N/eval-{id}/without_skill/outputs/
```

### 3-2. Choosing the baseline

| Situation | Baseline |
|------|----------|
| New skill | Run the same prompt with no skill |
| Improving an existing skill | The pre-edit skill version (snapshot it) |

### 3-3. Capturing timing data

Save `total_tokens` and `duration_ms` from the subagent completion notification **immediately**. This data is reachable only at notification time and cannot be recovered later.

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

## 4. Quantitative evaluation: assertion-based scoring

### 4-1. Writing assertions

When a deliverable is objectively verifiable, define assertions for automated scoring.

**Good assertion:**
- Decidable true/false objectively
- Named descriptively, so the result alone makes clear what was checked
- Tests the skill's core value

**Bad assertion:**
- Passes regardless of the skill (e.g. "output exists")
- Requires subjective judgment (e.g. "well written")

### 4-2. Make verification programmatic

If an assertion can be checked by code, write it as a script. Faster and more reliable than eyeballing, and reusable across iterations.

### 4-3. Beware non-discriminating assertions

An assertion that "passes 100% in both configurations" measures nothing about the skill's differential value. When you spot one, drop it or replace it with a more demanding assertion.

### 4-4. Scoring result schema

```json
{
  "expectations": [
    {
      "text": "Profit-margin column added",
      "passed": true,
      "evidence": "Confirmed 'profit_margin_pct' column in column E"
    },
    {
      "text": "Sorted descending by margin",
      "passed": false,
      "evidence": "Original order retained, no sort applied"
    }
  ],
  "summary": {
    "passed": 1,
    "failed": 1,
    "total": 2,
    "pass_rate": 0.50
  }
}
```

## 5. Specialist agents

Bringing in specialist-role agents during testing and evaluation improves quality.

### 5-1. Grader

Performs assertion-based scoring, and extracts verifiable claims from the deliverable to cross-check them.

**Role:**
- Pass/fail verdict per assertion, with evidence
- Extract factual claims from the deliverable and verify them
- Feedback on the eval itself (flag assertions that are too easy or ambiguous)

### 5-2. Comparator (blind comparison)

Anonymizes the two deliverables as A/B and judges quality without knowing which used the skill.

**When to use:** when you want a rigorous answer to "is the new version genuinely better?" Skippable in ordinary iteration.

**Judging criteria:**
- Content: accuracy, completeness
- Structure: organization, formatting, usability
- Overall score

### 5-3. Analyzer

Finds statistical patterns in the benchmark data:
- Non-discriminating assertions (pass in both configs → no discriminating power)
- High-variance evals (results swing run to run → unstable)
- Time/token trade-offs (skill raises quality but also cost)

## 6. Iterative improvement loop

### 6-1. Collecting feedback

Show the user the deliverable and gather feedback. Treat empty feedback as "no issues."

### 6-2. Improvement principles

1. **Generalize the feedback** — a narrow fix that only suits the test example is overfitting. Fix at the level of principle.
2. **Cut what doesn't earn its weight** — read the transcript; if the skill makes the agent do unproductive work, delete that part.
3. **Explain the why** — even when the user's feedback is terse, understand why it matters and fold that understanding into the skill.
4. **Bundle repeated work** — if every test run generates the same helper script, ship it in `scripts/` up front.

### 6-3. The iteration procedure

```
1. Edit the skill
2. Re-run all test cases in a fresh iteration-N+1/ directory
3. Present results to the user (compared against the previous iteration)
4. Collect feedback
5. Edit again → repeat
```

**Stop when:**
- the user is satisfied
- all feedback is empty (every deliverable is fine)
- there's no more meaningful improvement to make

### 6-4. The draft → review pattern

When editing a skill, write a draft, then **re-read it with fresh eyes** and improve. Don't try to nail it in one pass; run the draft-review cycle.

## 7. Description trigger verification

### 7-1. Writing trigger eval queries

Write 20 eval queries — 10 should-trigger + 10 should-NOT-trigger.

**Query quality bar:**
- Specific, natural sentences a real user would type
- Concrete details: file paths, personal context, column names, company names
- Mixed length, tone, and format
- Focus on **edge cases** rather than obvious matches

**Should-trigger queries (8–10):**
- The same intent phrased many ways (formal/casual)
- Cases that don't name the skill or file type but clearly need it
- Off-the-beaten-path use cases
- Cases that compete with another skill but where this one should win

**Should-NOT-trigger queries (8–10):**
- **Near-misses are the point** — similar keywords, but a different tool/skill is the right fit
- Obviously unrelated queries ("write a Fibonacci function") have no testing value
- Adjacent domains, ambiguous phrasing, overlapping keywords but a different context

### 7-2. Checking for collisions with existing skills

Confirm the new skill's description doesn't overlap the trigger territory of an existing skill:

1. Collect the descriptions of the existing skills
2. Confirm the new skill's should-trigger queries don't mistakenly trigger an existing one
3. On a collision, state the description's boundary conditions more precisely

### 7-3. Automated optimization (optional, advanced)

When description optimization is warranted:

1. Split the 20 eval queries Train (60%) / Test (40%)
2. Measure trigger accuracy with the current description
3. Analyze failures and generate an improved description
4. Pick the best description by the Test set (not the Train set — avoids overfitting)
5. Up to 5 iterations

> Run this as an automation script using `claude -p`. Token cost is high, so do it as a final step once the skill has stabilized.

## 8. Workspace structure

A directory layout to keep test/evaluation results organized:

```
{skill-name}-workspace/
├── iteration-1/
│   ├── eval-descriptive-name-1/
│   │   ├── eval_metadata.json
│   │   ├── with_skill/
│   │   │   ├── outputs/
│   │   │   ├── timing.json
│   │   │   └── grading.json
│   │   └── without_skill/
│   │       ├── outputs/
│   │       ├── timing.json
│   │       └── grading.json
│   ├── eval-descriptive-name-2/
│   │   └── ...
│   └── benchmark.json
├── iteration-2/
│   └── ...
└── evals/
    └── evals.json
```

**Rules:**
- Name eval directories **descriptively**, not numerically (e.g. `eval-multi-page-table-extraction`)
- Keep each iteration in its own directory (never overwrite a previous one)
- Don't delete `_workspace/` — it's the post-hoc verification and audit trail
