## 前提: 内部整合は証拠にならない

vibecoding では実装とテストを**同一プロセスが同じ誤解のもとで生成**するため、テストは実装の写像(トートロジー)になる。「全テストパス」「schema valid」「CI green」は虚偽の不在を意味しない。検証は必ず**リポジトリの外の世界**(実ツール実行、正典データ、独立再計算)と突き合わせる。

サブエージェントや過去セッションの「実行して確認しました」報告も内部整合の一種。追実行するまで `未検証` として扱う。

全 finding に認識論ラベルを付ける: `事実`(自分が実行・観察した) / `推測`(知識ベース。根拠を1行添える) / `未検証`(環境制約等で検証不能。合格扱いしない)。

## 虚偽の類型と検出ルール

| 型 | 名称 | 生成メカニズム | 検出ルール |
|----|------|----------------|-----------|
| 0 | 存在の捏造 | 実在しない API・部品型番・フラグ・引用の幻覚 | 一次ソースと突合。生成コマンドは実ツールで実行し `Unrecognized option` 系エラーの有無を見る |
| A | 形式準拠・意味違反 | 仕様の語彙は知っているが単位換算・符号化規則の適用を誤る(床除算、単位系、フィールドの文脈違い) | **独立実装で再計算・デコード**する。同じコードで encode→decode すると原理的に検出不能 |
| B | 成果物間の相互矛盾 | 成果物が別セッションで生成され、他の成果物を制約として読んでいない。各ファイル単体は自己整合 | 同一の量・能力を主張する成果物ペアを列挙し、ペアごとにクロス検算 |
| C | 名前だけの結合 | 「繋がって見せる」最小コストの応答が名前の複製。データフローが不在(dead UI / dead option) | 入力・選択を変えて**出力が変わるか**を観察する振る舞い検証。文字列 grep では原理的に検出不能 |
| D | 検証の演出 | 検証機構そのものが失敗しえない(grep テスト、単項チェックのみの validator)、または主張範囲 > 実行範囲 | mutation(実装を壊してテストが落ちるか)。「検証している」という主張と CI が実際に実行する範囲の diff |
| E | ドキュメントドリフト | docs が実装の追加に追随しない。従来型・軽微 | CLI/API 表面(--help、公開シンボル)と docs の diff |

型D は型0〜C を素通りさせる上位の虚偽であり、見つけたら最優先で報告する。

## 監査ステップ

### 1. 主張の棚卸し

README・docs・manifest・テスト名・コミットメッセージから「このリポジトリが主張していること」を列挙する。虚偽は主張と実体の差分なので、主張リストが監査のスコープになる。

### 2. 型0スクリーニング(存在確認)

最初に実施する。型番・API・依存・引用が実在するかを一次ソースと突合。ここで捏造が出るリポジトリは以降の全数値を疑う。

### 3. 外部突合

生成コマンド・生成バイナリ・生成設定を**実ツールに食わせる**。実行できるものは実行し、エラーの種類で判別する(引数パース通過後のランタイムエラーなら構文は本物)。既知の正典データ(標準のバイト列・定数)と照合できるものは照合する。

### 4. 独立再計算

JSON/manifest の合計値・マージン・タイミング・単位換算を**生データから自分で検算**する。ファイル内の total / margin フィールドを信用しない(validator が計算せず転記している可能性)。

<example>
<input>EDID manifest は「1080p60 対応」を主張。バイト列生成テストは全パス</input>
<output>
独立デコード: DTD の pixel clock = 148.5MHz(正典 VIC 16 バイト列と一致 = 型0/A なし)。
しかし range limits descriptor の max pixel clock = 0x0e × 10 = 140MHz < 148.5MHz。
`事実` EDID が自分自身の優先モードを禁止している。根因: `pixel_clock_khz // 10_000` の床除算
(10MHz 単位フィールドには切り上げが正解)。型A。checksum 有効・テスト全パスのまま製品として機能しない。
さらに「validator が整合を保つ」という docs の主張はこの矛盾を検出しない → 型D も併発。
</output>
</example>

### 5. クロス整合

同一の量を語る成果物ペアの表を作り、ペアごとに検算する。例: 帯域(エンドポイント宣言 vs 広告フォーマットの要求)、能力(要件 docs vs 部品の物理上限)、経路(ブロック図 vs 部品表)。「その機能を担う部品/コードがどこにも無い」形の欠落(エンコーダ不在等)もここで出る。

### 6. 振る舞い検証(型C)

UI の選択肢・CLI オプション・設定値を変えて、出力が実際に変わるかを観察する。名前が本体と共有されていても、差分が出なければ dead coupling。

### 7. 検証機構自体の監査(型D)

- テストが文字列 grep や存在チェックだけになっていないか。疑わしければ実装を意図的に壊してテストが落ちるか確認(mutation)
- CI が実行する範囲と README/docs が「検証済み」と主張する範囲の diff(例: `requires-python >=3.9` を主張して 3.12 でしかテストしない)
- validator がクロス整合(ステップ5)を検査しているか、単項チェック止まりか
- validator 自身が不正入力で契約通りエラーを返すか(クラッシュは契約不履行)

## 出力フォーマット

```markdown
# Falsehood Audit: <repo>
## 発見した虚偽
| 型 | ラベル | 内容 | 再現手順/根拠 |
## 虚偽なしと確認した項目(根拠つき)
| 項目 | 検証方法(どのルールか) | 結果 |
## 未検証(環境制約等)
<何が・なぜ検証できなかったか。合格に数えない>
```

「虚偽ゼロ」と報告してよいのは、型0〜Eの全ルールを実行した上で全て空振りした場合のみ。実行しなかったルールがあるなら未検証欄に残す。silence は合格ではない。

## アンチパターン

| パターン | なぜ失格 |
|----------|----------|
| テスト全パスを「本物」の根拠にする | テストは実装の写像。トートロジー |
| サブエージェントの実行報告を追実行せず採用 | 内部整合の一種。`未検証` のまま断言になる |
| 単項チェックの validator をクロス整合の証拠と誤読 | 型B はフィールド存在チェックを素通りする |
| encode したコードで decode して「一致」と報告 | 型A は同一実装の往復では原理的に出ない |
| 構文チェックだけで「動作確認済み」と報告 | 監査レポート自体が型D になる |
